# 网络深度与训练轮数检查

## 参考 Source_Code 配置

最早提供的 `Source_Code_0708` 中，三个已有算例的默认配置大致为：

| 算例 | 网络结构 | 默认 epoch |
|---|---:|---:|
| Navier-Stokes | `3 -> 20 x 8 -> 2` | `50000` |
| Linear Elasticity | `2 -> 50 x 4 -> 5` | `30000` |
| Heat Conduction | `2 -> 50 x 4 -> 1` | `20000` |

这些配置更偏“长时间训练的小/中等网络”。尤其 Navier-Stokes 使用较深但较窄的 MLP，通过流函数输出满足不可压约束。

## 当前 baseline pipeline 配置

当前 `scripts/run_experiment_pipeline.py` 的配置为：

| 算例 | 网络结构 | epoch | 判断 |
|---|---:|---:|---|
| Heat | `2 -> 64 x 4 -> 1` | `3000` | 宽度略大，深度接近，epoch 明显少于原始代码 |
| Burgers | `3 -> 64 x 4 -> 2` | `2500` | 新增算例，无原始对照；配置适中 |
| Navier-Stokes | `3 -> 64 x 4 -> 2` | `3000` | 比原始代码更宽但更浅，epoch 明显更少 |
| Linear Elasticity | `2 -> 64 x 5 -> 5` | `5000` | 比原始略宽略深，epoch 明显更少 |

结论：当前网络不是明显过深，epoch 也不是过多。相反，为了快速出图，当前 epoch 远少于原始代码，因此如果追求更高精度，应该增加训练时间或采用分阶段训练，而不是缩小网络。

## 修改建议

1. 保留 `64` 宽度作为当前统一基线，便于四类方程比较。
2. Heat 和 Burgers 可以继续用 `64 x 4`。
3. Navier-Stokes 若追求更好压力场，可考虑改为 `64 x 6` 或 `32 x 8`，并增加压力参考点/零均值约束。
4. Linear Elasticity 当前主要问题是损失尺度差异，而不是网络过深；更应做无量纲化、分项权重或经验公式消融。
5. 原始源码 epoch 很大，实际研究中建议先用当前 baseline pipeline 快速筛选，再对关键组别长训。

## 关于可视化“都像热点”的说明

当前四类数据都是单位方形上的制造解，且大量变量由 `sin`、`cos` 组合构造。使用同一种 `imshow + viridis` 色图时，确实容易看起来都是中心或边界附近的“热点”。这不代表数据生成错了，主要原因是：

- 几何域完全相同，都是 `[0,1] x [0,1]`；
- 制造解故意选了平滑三角函数，便于自动微分和残差验证；
- `fields.png` 默认展示的是同一时间切片，并且所有变量都被画成热图；
- 不同物理场的差异更多体现在变量含义、符号、导数关系和 PDE 残差，而不是几何外观。

为此新增了 `scripts/visualize_dataset.py`，它直接画数据集真值场，并对正负变量自动使用 `coolwarm`，比训练预测图更适合检查数据集差异。
